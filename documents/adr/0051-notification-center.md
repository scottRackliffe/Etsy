# ADR-051: Notification center

## Status

Accepted

## Date

2026-05-24

## Context

Toasts disappear after 5 seconds and are gone forever. If a user steps away from the screen or is focused on another part of the page, they miss important system events like sync completions, backup failures, or validation errors. Users need a persistent log of system events they may have missed.

## Decision

Add a notification center accessible via a bell icon in the app header. Notifications are ephemeral (session-scoped, stored in localStorage) and provide a rolling history of system events.

### UI location

- Bell icon (🔔) in the `AppHeader` component, right side, before any user/settings controls.
- Unread count badge: small circle with number overlaid on the bell icon (red background, white text). Hidden when count is 0.
- Click bell → dropdown panel (320px wide, max 480px tall, scrollable) anchored to the bell icon.

### Notification types

| Type      | Visual indicator | Use cases                                                       |
| --------- | ---------------- | --------------------------------------------------------------- |
| `success` | Green dot        | Record saved, import complete, sync succeeded, backup completed |
| `error`   | Red dot          | Sync failed, save failed, network errors, Etsy API errors       |
| `warning` | Yellow dot       | Partial import (some rows skipped), connection lost, stale data |
| `info`    | Blue dot         | Etsy connection status change, reminder, system message         |

### Notification sources

All toasts generated by the app are simultaneously stored as notifications. Additional sources:

- Etsy sync results (success with counts, or failure with reason)
- Backup completion or failure
- Etsy connection/disconnection events
- Mutation queue replay results (ADR-050)
- CSV import results (ADR-047)
- Validation failures that affect multiple records

### Data model

Stored in `localStorage` under key `esm_notifications`:

```typescript
interface Notification {
  id: string; // UUID v4
  type: "success" | "error" | "warning" | "info";
  message: string; // Human-readable message (same as toast text)
  timestamp: string; // ISO 8601
  read: boolean; // default false
  action?: {
    label: string; // e.g., "View order", "Reload", "Open config"
    url: string; // Internal route, e.g., "/sales?orderId=123"
  };
}
```

- **NOT stored in the database** — notifications are ephemeral and session-scoped. The `activity_log` table (ADR-037) serves as the persistent audit trail.
- Maximum stored: 50 notifications. When a new notification would exceed 50, the oldest read notification is removed. If all are unread, the oldest is removed regardless.

### Dropdown panel behavior

- Header: "Notifications" title + "Mark all read" text button (right-aligned)
- List: notifications sorted by timestamp descending (newest first)
- Each notification row shows:
  - Type dot (colored circle, left)
  - Message text (truncated to 2 lines with ellipsis)
  - Relative timestamp ("2m ago", "1h ago", "Yesterday")
  - Action button (if present) — small text link on the right
- Clicking a notification:
  - Marks it as read
  - If it has an `action.url`: navigates to that URL
- Empty state: "No notifications" centered text
- Close: clicking outside the dropdown or pressing `Escape` closes it

### Mark as read

- Clicking a notification marks it as read (dot becomes grey/faded).
- "Mark all read" button marks all notifications as read and resets the badge count to 0.
- Opening the dropdown does NOT auto-mark notifications as read.

### Auto-clear

- On page load (app initialization): remove all notifications older than 24 hours.
- Clearing is silent — no user notification about cleared notifications.

### Integration with Toast component

The existing `Toast` component (ADR-028) is extended:

- Every toast that is shown also creates a notification via a shared `addNotification()` function from a `useNotifications` context provider.
- Toasts continue to auto-dismiss after 5 seconds as before.
- The notification persists in the notification center regardless of toast dismissal.

### No external notifications

- No push notifications (no service worker push).
- No email notifications.
- No sound alerts.
- Purely a local UI feature.

## Consequences

- **Positive**: Users never miss important system events. Provides accountability for background operations (sync, backup). Action buttons enable quick navigation to relevant records. Lightweight — no server storage or API endpoints needed.
- **Negative**: Limited to 50 items and 24 hours — not a long-term audit trail (that's activity_log's job). localStorage can be cleared by the browser. No cross-tab synchronization (notifications created in one tab won't appear in another until page reload).

## Notes

- Cross-references: ADR-028 (Toast component — notifications extend toasts), ADR-037 (activity log — the persistent, DB-backed audit trail; notifications are the ephemeral UI counterpart), ADR-050 (network loss — failed replay items appear as notifications), ADR-035 (deep links — notification action URLs use the same deep-link format)
- Implementation note: use a React Context provider (`NotificationProvider`) wrapping the app layout to provide `addNotification`, `markRead`, `markAllRead`, `notifications`, and `unreadCount` to all components.
- The 50-notification limit and 24-hour TTL are chosen to keep localStorage usage minimal (~50KB worst case).
